# Спецификация для модели: JSON -> Паспорт изделия

Цель: обеспечить максимально точную и воспроизводимую генерацию паспорта изделия по входному `normalized_drawing_json`, сформированному из DXF, PDF или изображения.

## 1. Область применения

- Вход: JSON, сформированный пайплайном проекта (`InputRouter` + DXF/raster ingest).
- Выход: технический паспорт изделия в строго заданной структуре.
- Язык вывода: русский.

## 2. Ключевые принципы

- Использовать только данные из JSON.
- Не выдумывать значения, ГОСТ, допуски, материал, твердость, размеры.
- Если данных нет или они неоднозначны, писать: `Не указано в чертеже`.
- Сохранять численные значения в исходном виде (без произвольного округления).
- При конфликте значений указывать это в разделе примечаний.

## 3. Ожидаемый входной JSON

Основной формат в первой версии:

- `source`
- `preview`
- `drawing_facts`
- `ocr_blocks`
- `vision_blocks`
- `semantic_candidates`
- `evidence`
- `legacy_summary`

JSON может содержать:

- `source`:
  - `input_type`
  - `file_name`
  - `original_path`
  - `mime_type`
  - `size_bytes`
  - `checksum_sha256`

- `preview`:
  - `path`
  - `format`
  - `width`
  - `height`
  - `dpi`

- `drawing_facts`:
  - `units`
  - `entity_counts`
  - `layers`
  - `dimensions`
  - `bounding_box`
  - `extracted_texts`
  - `geometry`
  - `feature_collection`

- `ocr_blocks`:
  - массив OCR-фрагментов с `text`, `raw_text`, `bbox`, `confidence`, `page`

- `vision_blocks`:
  - массив визуальных блоков (`title_block`, `view`, `dimension_zone`, `note_block`, ...)

- `semantic_candidates`:
  - предзаполненный слой кандидатов для LLM

- `evidence`:
  - словарь полей -> список источников

- `legacy_summary`:
  - DXF-совместимый слой для обратной совместимости

- `feature_collection` (DXF2JSON-style):
  - `type: "FeatureCollection"`
  - `features[]` с `properties` и `geometry`
  - `properties.ENTITIES` in `{LWPOLYLINE, LINE, CIRCLE, MTEXT, POINT, INSERT, ...}`

## 4. Приоритет источников данных

При извлечении фактов для паспорта использовать приоритет:

1. `semantic_candidates.*.evidence`
2. Атрибуты `INSERT` (блоковые атрибуты `TAG: VALUE`)
3. `MTEXT.properties.LaNote`
4. `drawing_facts.extracted_texts`
5. `legacy_summary.designation_guess` / `legacy_summary.title_guess`
6. `drawing_facts.dimensions`, `drawing_facts.bounding_box`, `drawing_facts.entity_counts`

Если источник более высокого приоритета противоречит нижнему, брать более высокий и фиксировать конфликт в `ПРИМЕЧАНИЯ`.

## 5. Нормализация фактов перед генерацией

Модель должна сначала сформировать внутренний фактический слой:

- `product_name`
- `designation`
- `units`
- `material_hardness`
- `overall_dimensions`
- `geometry_facts[]`
- `gdt_facts[]`
- `notes_facts[]`
- `missing_fields[]`
- `conflicts[]`

Правила:

- `product_name`: из `semantic_candidates.product_name`; при отсутствии из `legacy_summary.title_guess` и/или явных текстов.
- `designation`: из `semantic_candidates.designation`; при отсутствии из паттернов в текстах/атрибутах.
- `overall_dimensions`: в приоритете `semantic_candidates.overall_dimensions`; иначе явные размерные подписи; иначе допустимо использовать `drawing_facts.bounding_box` с пометкой.
- `material_hardness`: только из явных текстов про материал/твердость.
- `gdt_facts`: только из явно распознанных обозначений/текстов ГДТ.

## 6. Формат выходного паспорта

Строго 4 раздела:

1. `ОБЩИЕ ДАННЫЕ`
2. `ГЕОМЕТРИЯ (ЧИСТОВАЯ)`
3. `ГДТ`
4. `ПРИМЕЧАНИЯ`

### 6.1 Раздел 1. ОБЩИЕ ДАННЫЕ

Обязательные поля:

- Тип
- Обозначение
- Габариты (Макс)
- Материал/Твердость

Если поле не подтверждено данными JSON:
- значение = `Не указано в чертеже`.

### 6.2 Раздел 2. ГЕОМЕТРИЯ (ЧИСТОВАЯ)

Структура:

- Наружный контур
- Внутренняя система
- Спец. элементы

Правила:

- перечислять только подтвержденные элементы;
- избегать общих фраз;
- не интерпретировать геометрию в технологические выводы без текстового подтверждения.

### 6.3 Раздел 3. ГДТ

- включать только явно найденные допуски геометрии;
- если нет подтверждения, выводить: `Не указано в чертеже`.

### 6.4 Раздел 4. ПРИМЕЧАНИЯ

Включать:

- неоднозначности и конфликты;
- поля, заполненные по косвенным данным (например, bbox);
- список критичных пропусков.

## 7. Запреты для модели

Модель НЕ должна:

- придумывать ГОСТ, марку материала, твердость;
- выводить размеры, которых нет во входных данных;
- подменять отсутствующие ГДТ общими шаблонными формулировками;
- менять единицы измерения без явного указания.

## 8. Рекомендуемый формат машинного ответа (до рендера в DOCX)

Рекомендуется сначала формировать структурированный JSON паспорта:

```json
{
  "product_name": "",
  "designation": "",
  "units": "",
  "sections": {
    "general": {
      "type": "",
      "dimensions_max": "",
      "material_hardness": ""
    },
    "geometry": {
      "outer_contour": [],
      "inner_system": [],
      "special_elements": []
    },
    "gdt": [],
    "notes": []
  },
  "missing_fields": [],
  "conflicts": [],
  "evidence": {
    "product_name": [],
    "designation": [],
    "dimensions_max": [],
    "material_hardness": [],
    "gdt": []
  }
}
```

Где `evidence` — ссылки на источник в JSON (например, `feature handle`, `layer`, ключ атрибута).

## 9. Критерии качества результата

Минимум для `PASS`:

- заполнены `product_name` и `designation` либо явно помечены как отсутствующие;
- нет выдуманных значений;
- все критичные поля имеют `evidence` или пометку `Не указано в чертеже`;
- разделы паспорта строго соответствуют шаблону из 4 разделов.

`WARN`:

- есть неоднозначности/конфликты;
- часть критичных полей отсутствует.

`FAIL`:

- обнаружены вымышленные данные;
- нарушена структура паспорта;
- отсутствует трассируемость к источникам JSON.

## 11. Практический совет по точности

Для максимальной достоверности сначала генерировать не markdown, а структурированный JSON паспорта, валидировать его, и только потом рендерить в DOCX/MD.

## 12. Словарь обозначений (обязателен для модели)

Модель должна трактовать поля строго по этой легенде.

### 12.1 `Feature.properties`

- `ENTITIES`:
  - `LWPOLYLINE` — ломаная/контур;
  - `LINE` — одиночный отрезок;
  - `CIRCLE` — окружность (в этом экспорте как полигональная аппроксимация);
  - `MTEXT` — многострочная текстовая заметка;
  - `POINT` — точка;
  - `INSERT` — вхождение блока (часто содержит критичные атрибуты изделия).
- `LayerName` — слой чертежа (может определять класс данных: размеры, штамп, техтребования).
- `Handle` — уникальный идентификатор объекта в DXF.
- `laCouleur` — цвет (`BYLAYER` или код).
- `Link` — ссылка из XDATA (если есть).
- `LaNote` — текст примечания для `MTEXT`.
- `leBloc` — имя блока для `INSERT`.
- остальные ключи в `INSERT` — атрибуты блока `TAG: VALUE`, потенциально самые важные для паспорта.

### 12.2 `Feature.geometry`

- `LineString.coordinates` — список точек `[x, y]` в порядке задания в DXF.
- `Polygon.coordinates[0]` — внешний контур.
- `Point.coordinates` — координата вставки/точки/текста.

## 13. Словарь инженерных символов в тексте

При разборе текстов (`LaNote`, атрибуты блока, extracted_texts) применять:

- `Ø` / `⌀` / `DIA` -> диаметр.
- `R` -> радиус.
- `±` -> симметричный допуск.
- `H7`, `H9`, `h11`, `e8` и т.п. -> поле посадки/допуска.
- `x`, `×` между числами -> произведение размеров (например, фаска/габарит).
- `HRC` -> твердость.
- `ГОСТ` -> стандарт.
- `\P` (если осталось) -> перенос строки.

Если символ есть, но числовой контекст не распознан, переносить фрагмент в примечания без домысливания.

## 14. Обязательный алгоритм чтения JSON моделью

Перед генерацией паспорта модель обязана выполнить шаги:

1. Пройти `feature_collection.features` и сгруппировать по `ENTITIES`.
2. Собрать каталог `INSERT` атрибутов (источник №1).
3. Собрать все `MTEXT.LaNote` (источник №2).
4. Собрать числовые и текстовые паттерны из заметок/атрибутов.
5. Сопоставить найденные факты с полями паспорта.
6. Для каждого поля паспорта сформировать `evidence` (минимум один источник) или отметить `Не указано в чертеже`.
7. Только после этого рендерить финальные 4 раздела паспорта.

## 15. Правила уверенности (confidence)

Для каждого критичного поля:

- `high`: есть явное значение + источник (`INSERT`/`MTEXT`) + нет конфликтов;
- `medium`: значение получено косвенно (например, bbox);
- `low`: неоднозначность или конкурирующие значения.

Поля `low` всегда отражать в `ПРИМЕЧАНИЯ`.

## 16. Минимальные требования к "идеальному" паспорту

Паспорт считается максимально корректным, если:

- все критичные поля заполнены либо честно помечены как отсутствующие;
- нет выдуманных данных;
- у каждого критичного поля есть `evidence`;
- модель явно указывает источники в JSON (`Handle`, `LayerName`, `ENTITIES`).

## 17. Архитектура трех моделей

`normalized_drawing_json` проектируется как общий контракт между тремя моделями:

1. Vision-модель:
   - заполняет `vision_blocks`
   - может дополнять `evidence` и `semantic_candidates`
2. OCR-модель:
   - заполняет `ocr_blocks`
   - нормализует текстовые значения, размеры, обозначения
3. LLM:
   - читает `drawing_facts`, `ocr_blocks`, `vision_blocks`, `semantic_candidates`, `evidence`
   - формирует `semantic_passport_json`
   - затем рендерит финальный паспорт
